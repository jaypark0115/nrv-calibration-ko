4. AXI-to-APB Bridge 설계와 검증

최종 프로젝트는 앞에서 배운 AXI와 APB를 직접 연결하는 AXI-to-APB Bridge 구현이었습니다. Bridge는 서로 다른 bus protocol 사이의 번역기입니다. AXI 쪽에서는 channel별 VALID/READY handshake로 요청이 들어오고, APB 쪽에서는 setup-enable phase로 register access가 진행됩니다. 저는 이 사이를 Verilog FSM으로 연결하고, 4개 register를 가진 APB slave와 testbench로 write/read 동작을 검증했습니다.

이 페이지의 범위 Axi2Apb RTL, APB slave register, top 연결, task 기반 testbench, write/read waveform, 최종 PASS 로그를 설명합니다.

4.1 왜 AXI-to-APB Bridge를 구현했는가

SoC 안에서는 모든 블록이 같은 bus를 쓰지 않습니다. 고성능 master나 interconnect는 AXI처럼 채널이 분리된 protocol을 사용할 수 있고, 주변장치 register는 APB처럼 단순한 control bus에 붙는 경우가 많습니다. AXI-to-APB Bridge는 이 둘 사이에서 AXI 요청을 받아 APB register access로 바꿔 주는 블록입니다.

예를 들어 software가 AXI 경로로 어떤 register에 값을 쓰려고 하면, bridge는 AXI write address와 write data를 받은 뒤 APB의 PADDR, PWRITE, PWDATA, PSEL, PENABLE로 변환해야 합니다. read도 마찬가지로 AXI read address를 APB read transfer로 바꾸고, APB slave에서 읽은 값을 AXI R channel로 돌려줍니다. 이 프로젝트는 software가 bus를 통해 hardware register에 접근하는 가장 작은 end-to-end loop를 직접 구현한 경험이었습니다.

4.2 FSM으로 protocol 변환 순서를 설계

Bridge 구현의 중심은 FSM이었습니다. write 쪽은 AXI AW 요청을 받고, W data를 받은 뒤, APB setup-enable transfer를 만들고, 마지막에 AXI B response를 반환하는 순서로 구성했습니다. read 쪽은 AXI AR 요청을 받은 뒤, APB read setup-enable transfer를 만들고, APB에서 읽은 data를 AXI R channel로 반환하는 순서로 구성했습니다.

01AXI 요청 수신

AW/W 또는 AR channel에서 주소와 data 또는 read 요청을 받습니다.

02Bridge FSM

write FSM과 read FSM이 APB transfer 순서에 맞게 상태를 전환합니다.

03APB 제어신호 생성

PSEL, PENABLE, PWRITE, PADDR, PWDATA를 setup/enable phase에 맞춥니다.

04AXI 응답 반환

write는 B response, read는 R data와 RLAST를 AXI 쪽으로 돌려줍니다.

4.3 최종 프로젝트에서 만든 블록

최종 결과는 세 개의 RTL 블록과 하나의 testbench로 정리할 수 있습니다. Axi2Apb.v는 AXI slave 입력을 받아 APB master 신호를 만드는 bridge입니다. ApbSlave.v는 0x0, 0x4, 0x8, 0xC 주소에 대응하는 4개 32-bit register를 제공합니다. Prj_Axi_Top.v는 bridge와 slave를 연결하는 상위 모듈이고, TbTop_Prj_Axi.v는 AXI master 역할을 하며 write/read transaction을 생성합니다.

Axi2Apb

Protocol conversion

AXI AW/W/B, AR/R 흐름을 APB setup-enable transfer로 변환하는 write/read FSM을 구현했습니다.

ApbSlave

4-register target

4개 register에 대해 write commit과 read data mux를 구현해 bridge가 접근할 목적지 block을 만들었습니다.

Testbench

반복 검증

task 기반 write/read 시나리오로 4개 주소를 쓰고 다시 읽어 기대값과 비교했습니다.

4.4 Waveform과 log로 확인한 최종 동작

검증은 파형만 보는 방식으로 끝내지 않고, testbench에서 write/read task를 만들어 반복 가능한 검증 루틴으로 구성했습니다. 각 register에 값을 쓴 뒤 다시 readback하고, handshake timeout, response 오류, last 조건을 확인했습니다. 최종 로그에서는 PASS: 4REG write/read + timing OK를 확인했습니다.

4.5 이 실습이 이후 시스템 설계와 연결되는 지점

이 프로젝트를 통해 bus protocol을 이론이 아니라 RTL 구조와 검증 루틴으로 다뤄 보았습니다. 특히 센서나 FPGA가 포함된 비전 시스템에서는 알고리즘 이전에 register 설정, data path 제어, host와 하드웨어 사이의 통신 구조가 안정적으로 동작해야 합니다. APB/AHB/AXI Lab과 AXI-to-APB Bridge 경험은 이후 이벤트센서 시스템이나 FPGA 기반 비전 가속기를 다룰 때, 하드웨어 블록이 소프트웨어와 어떤 방식으로 연결되고 검증되어야 하는지 이해하는 기반이 되었습니다.
