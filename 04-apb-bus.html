1. APB Control Bus

APB는 SoC 안의 주변장치 register를 읽고 쓰기 위한 단순한 control bus입니다. 숭실대 인턴에서는 APB를 단순 개념으로만 외우지 않고, register block을 직접 만들고, SRAM 접근과 interrupt 구조로 확장하며, SimVision 파형으로 PSEL/PENABLE/PREADY가 어떤 순서로 동작하는지 확인했습니다.

이 페이지의 범위 APB가 왜 control bus로 쓰이는지, Lab2~4에서 어떤 RTL/TB 실습을 했는지, 파형에서 무엇을 확인했는지 정리합니다.

1.1 APB를 먼저 배운 이유

SoC에는 성격이 다른 데이터 이동이 섞여 있습니다. 이미지 버퍼나 메모리처럼 큰 데이터를 빠르게 옮겨야 하는 경로도 있고, 센서 모드 설정, interrupt mask, status bit 확인처럼 register 몇 개만 읽고 쓰면 되는 경로도 있습니다. APB는 후자에 맞춰진 bus입니다. 복잡한 burst나 outstanding transaction보다, 정해진 주소의 register를 안정적으로 제어하는 데 초점을 둡니다.

수업자료에서는 APB를 “HW control & status monitor”를 위한 bus로 다뤘고, Lab2에서 바로 register interface를 구현했습니다. 그래서 이 페이지의 핵심도 APB 신호 이름을 암기하는 것이 아니라, software가 주소를 보내면 hardware register가 어떻게 선택되고 갱신되는지 이해하는 데 있습니다.

처음 보는 사람을 위한 설명

APB는 주변장치의 설정 버튼을 누르는 통로에 가깝습니다

예를 들어 카메라 센서의 설정값을 바꾸거나, 어떤 interrupt를 켤지 정하거나, 하드웨어 블록의 상태값을 읽는 일은 대용량 데이터 전송이 아니라 register 제어입니다. APB는 이런 제어 작업을 간단한 타이밍으로 처리하게 해 줍니다.

1.2 Setup phase와 Enable phase

APB transfer는 두 단계로 이해하면 됩니다. 첫 번째 setup phase에서는 PSEL을 올리고 주소, write 여부, write data를 준비합니다. 두 번째 enable phase에서는 PENABLE을 올려 실제 전송을 진행합니다. 이때 slave가 PREADY를 올리면 transfer가 완료됩니다. write라면 register가 갱신되고, read라면 PRDATA로 register 값이 반환됩니다.

PSEL
이번 transfer에서 해당 APB slave가 선택되었음을 나타냅니다.
PENABLE
setup 이후 실제 enable phase에 들어갔음을 나타냅니다.
PREADY
slave가 전송을 끝낼 준비가 되었는지 알려 줍니다. ready가 늦으면 transfer가 기다립니다.
PADDR/PWRITE/PWDATA/PRDATA
주소, write/read 방향, write data, read data를 담당합니다.

1.3 APB 계열 Lab에서 한 일

APB를 배운 뒤에는 Lab2에서 register access를 먼저 구현했습니다. APB address가 0x00, 0x04, 0x0C 같은 register 위치를 가리키면, RTL 내부에서 write enable 또는 read enable을 만들고 register 값을 갱신하거나 반환하는 구조입니다. 이 단계에서 주소 decode, write commit 조건, read data mux, PREADY 생성 조건을 직접 확인했습니다.
