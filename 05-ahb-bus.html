2. AHB Pipelined Bus

AHB는 APB보다 더 큰 SoC 내부 연결을 이해하기 위해 다룬 bus입니다. APB가 주변장치 register 제어에 적합했다면, AHB는 master, decoder, mux, slave가 있는 system bus 구조와 address/data phase가 겹쳐 흐르는 pipeline timing을 보여 줍니다. 숭실대 인턴에서는 Lab5에서 AHB register access를 구현하고, Lab6에서 AHB-to-SRAM 접근으로 확장했습니다.

이 페이지의 범위 AHB topology, address/data phase, HREADY/HRESP, Lab5 register access, Lab6 SP-SRAM access를 설명합니다.

2.1 APB 다음에 AHB를 배운 이유

APB를 통해 register를 어떻게 읽고 쓰는지 배운 뒤, 수업은 AHB로 넘어갔습니다. 이유는 SoC에서 모든 블록이 APB처럼 단순한 제어만 하는 것은 아니기 때문입니다. 여러 master와 slave가 연결되고, 주소 decode와 data return이 분리되며, ready 신호가 전체 timing을 제어하는 구조를 보려면 AHB 같은 system bus를 봐야 합니다.

수업 현장에서는 AHB와 AXI의 사용 조건도 비교했습니다. AHB는 구조가 비교적 단순하고 단일 master나 내부 IP 연결에 적합한 반면, AXI는 더 높은 성능과 확장성을 목표로 합니다. 이 구분을 먼저 잡아야 AXI-to-APB Bridge가 왜 필요한지도 이해할 수 있습니다.

2.2 Master, decoder, mux, slave 구조

AHB에서 master는 주소와 제어 정보를 내보내고, decoder는 주소 범위에 따라 어떤 slave가 선택될지 결정합니다. 선택된 slave는 data와 response를 반환하고, mux는 여러 slave 중 현재 선택된 slave의 값을 master 쪽으로 돌려보냅니다. 따라서 AHB는 하나의 register만 보는 것이 아니라, 주소가 slave select로 바뀌고 response가 다시 master로 돌아오는 전체 경로를 봐야 합니다.

Master

요청 시작

주소, read/write 방향, transfer type을 발생시킵니다.

Decoder

주소 해석

주소 범위를 보고 어떤 slave가 선택될지 결정합니다.

Slave / Mux

응답 반환

선택된 slave의 data와 response를 master로 돌려보냅니다.

2.3 Address phase와 Data phase

AHB의 핵심은 address phase와 data phase가 분리되어 있다는 점입니다. 어떤 cycle에서 HADDR, HTRANS, HWRITE 같은 주소/제어 신호가 들어오면, 실제 data 처리는 다음 data phase에서 진행됩니다. 동시에 다음 transfer의 address phase가 시작될 수 있기 때문에, 파형을 볼 때 현재 보이는 data가 어느 address와 대응되는지 phase 단위로 추적해야 합니다.

HREADY는 다음 phase로 넘어가도 되는지를 결정합니다. HREADY가 낮으면 bus는 hold되고, 다음 주소 phase도 진행되지 않습니다. Lab6에서는 synchronous SRAM read 특성 때문에 read 시점에 wait가 필요했고, 이때 HREADY가 bus timing을 어떻게 멈추는지 확인했습니다.

2.4 AHB 계열 Lab에서 한 일

Lab5에서는 AHB slave interface를 가진 register block을 구현했습니다. address phase에서 HSEL, HTRANS, HWRITE, HADDR를 latch하고, data phase에서 register A/B write 또는 register C read를 수행하는 구조입니다. APB보다 중요한 점은 write/read enable이 현재 cycle의 raw address가 아니라, 한 phase 앞에서 latch된 주소와 제어 정보를 기준으로 만들어진다는 점이었습니다.

Lab6에서는 같은 AHB interface를 SP-SRAM 접근으로 확장했습니다. write는 바로 SRAM에 기록할 수 있어 zero-wait로 처리했고, read는 synchronous SRAM 특성상 data가 한 클럭 뒤에 나오기 때문에 one-wait로 처리했습니다. 이를 통해 bus protocol과 memory timing을 맞추는 일이 단순 기능 구현과 다르다는 점을 확인했습니다.

Lab5

AHB register access

HSEL/HTRANS/HWRITE/HADDR를 latch하고, data phase에서 register write/read를 수행했습니다.

Lab6

AHB to SP-SRAM

SRAM write/read timing을 AHB HREADY와 연결해 zero-wait write, one-wait read를 검증했습니다.
